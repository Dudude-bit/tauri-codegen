# Changelog

All notable changes to `tauri-ts-generator` are listed here. Format
loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/);
versioning follows [SemVer](https://semver.org/).

## 2.0.3 — 2026-05-10

### Fixed

- Macro-generated `#[tauri::command]` functions (e.g. items emitted by
  a `macro_rules!` that stamps out N typed commands) are now picked up
  from `cargo expand` output. Previously the parser keyed off the
  `#[tauri::command]` attribute, which is consumed by procedural-macro
  expansion — leaving the bare `fn` body and a sibling
  `pub use __cmd__name;` re-export that the old parser ignored. A new
  pass collects every `__cmd__X` name from `use` items and matches
  them back to function definitions.
- `parse_commands` previously recursed only one level into `Item::Mod`
  and ignored `Item::Impl` blocks inside modules entirely — commands
  in patterns like `mod a { mod b { #[tauri::command] fn x() {} } }`
  or `mod a { impl Foo { #[tauri::command] fn bar() {} } }` were
  silently dropped. The walker now fully recurses; eight new tests pin
  the contract.
- Windows CI: four `config::tests` tests panicked with
  `too few unicode value digits, expected unicode hexadecimal value`
  because the test config snippets interpolated `tempdir().path()` —
  which on Windows is a backslash path — into a TOML basic string
  (`"{}"`). Backslashes are escape characters in basic strings; `\U`
  triggers Unicode escape processing. Switching to TOML literal
  strings (`'{}'`) takes the path verbatim and fixes Windows runs.

### Changed

- Added public `parse_expanded_commands(content, source_file)` to the
  `parser` module. It complements the existing `parse_commands` and is
  what the pipeline uses internally when `use_cargo_expand = true`.
  Removed two single-use private wrappers (`parse_command_fn` and
  `parse_command_method`) whose `Option<TauriCommand>` return type
  could never be `None`; callers now use `parse_command_from_signature`
  directly via a shared `walk_for_commands` helper.

### Known limitations

- For commands generated by a `macro_rules!` that pass `rename_all`
  through `#[tauri::command(rename_all = "...")]`, the value is
  **unrecoverable** from `cargo expand` output: Tauri 2.x emits only
  `pub use {__cmd__X, __tauri_command_name_X}` on the lib side; the
  body that bakes the renaming in is generated *on the consumer side*
  by `generate_handler!` (a different crate), and is not part of the
  expanded output the generator parses. Write the command directly in
  source if `rename_all` is required.

## 2.0.1 / 2.0.2 — 2026-05-10

Initial (incomplete) attempts at the macro-generated-commands fix. The
v2.0.1 patch re-ran `parse_commands` on cargo-expand output, but
procedural-macro expansion had already stripped the
`#[tauri::command]` attribute it filtered on, so nothing was found.
The v2.0.2 patch added the `__cmd__` marker recognition but kept two
parallel walkers. Both releases are superseded by v2.0.3 — upgrade
directly.

## 2.0.0

First release on the public repository (`Dudude-bit/tauri-codegen`).
See git history for prior changes from the `tauri-ts-generator-derive`
era.
